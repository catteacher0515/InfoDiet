# InfoDiet 联调与演示 SOP

## 1. 目标

这份文档用于跑通当前项目的真实演示链路，重点不是“把所有接口都点一遍”，而是按产品视角展示这套系统已经具备的能力：

1. 登录与角色区分
2. 用户工作台能力
3. 管理区用户与订阅治理
4. 运维区任务、失败推送、告警联动

---

## 2. 启动前提

### 后端

- 地址：`http://127.0.0.1:8123/api`
- 启动命令：

```bash
./mvnw spring-boot:run
```

### 中间件

当前项目启动依赖两个中间件：

1. Redis
2. RabbitMQ

RabbitMQ 采用默认本地配置：

- host：`localhost`
- port：`5672`
- username：`guest`
- password：`guest`

如果 RabbitMQ 没启动，推送入队链路会失败。

### 前端

- 地址：`http://127.0.0.1:5173`
- 启动命令：

```bash
cd frontend
pnpm dev
```

### 当前本地常见问题

如果新增接口已经写好了，但访问仍然返回 `404`，优先排查是不是旧进程占用了 `8123`：

```bash
lsof -iTCP:8123 -sTCP:LISTEN -n -P
kill <PID>
./mvnw spring-boot:run
```

## 4. 推荐演示顺序

### 第一段：管理员登录

访问：

- `http://127.0.0.1:5173/login`

使用管理员账号登录后，默认会进入：

- `/admin/dashboard`

这一段要讲的重点是：

- 这是一个带角色区分的系统，不是只有接口没有后台
- 管理区、用户区、运维区已经被拆开

---

### 第二段：管理区演示

依次进入：

1. `/admin/dashboard`
2. `/admin/users`
3. `/admin/subscriptions`

#### 在 `/admin/users` 重点演示

1. 创建用户
2. 点击某个用户的“查看”
3. 查看用户详情
4. 查看该用户的订阅信息
5. 启用 / 禁用用户

这一段重点说明：

- 系统已经不是“固定写死抓内容”
- 用户维度已经存在，并且可以治理用户状态
- 被禁用用户不会继续参与匹配和推送，但历史数据会保留

---

### 第三段：运维区演示

依次进入：

1. `/ops/dashboard`
2. `/ops/tasks`
3. `/ops/push-failures`
4. `/ops/alerts`

#### 在 `/ops/tasks` 重点演示

1. 直接触发任务
2. 按任务类型筛选
3. 按状态筛选
4. 查看单条任务详情
5. 对某条任务执行“重跑”

这一段重点说明：

- 任务不只是定时器黑盒
- 后台已经能观察任务运行结果
- 失败后可以回看详情，也可以直接重跑

#### 在 `/ops/push-failures` 重点演示

1. 按关键字筛选失败推送
2. 按重试次数筛选
3. 翻页
4. 查看关联信息
5. 单条重试
6. 批量重试

这一段重点说明：

- 失败推送不是静态列表
- 已经具备运维处理动作
- 可以把失败推送、内容、告警三者串起来看

#### 在 `/ops/alerts` 重点演示

1. 查看待处理告警
2. 标记已发送
3. 发送到飞书

这一段重点说明：

- 系统已经不是“失败了没人知道”
- 告警链路已经接入真实渠道

---

### 第四段：普通用户工作台演示

退出后用普通用户登录，进入：

- `/workspace/dashboard`

再依次进入：

1. `/workspace/subscriptions`
2. `/workspace/content`
3. `/workspace/pushes`

这一段重点说明：

- 管理员和普通用户看到的是不同工作台
- 普通用户能维护自己的订阅
- 普通用户能查看匹配到的内容和推送记录

## 5. 推荐的“完整故事线”

如果你是给别人演示，不建议只是一个页面一个页面点。

更自然的讲法是：

1. 这是一个“信息节食”系统，不是刷榜工具
2. 用户先配置订阅和来源
3. 系统按订阅去抓内容，而不是无脑抓全网
4. 内容入库后做匹配和推送
5. 管理员可以管理用户和订阅结构
6. 运维可以查看任务、失败推送、告警，并做重跑和重试

这样讲，整个项目会更像产品，而不是接口拼盘。

## 7. 当前阶段的边界

当前演示可以支撑“这是一个完整产品雏形”，但还不是最终完整版。

当前边界包括：

1. 推送渠道目前仍以飞书为主
2. 页面是管理后台风格，不是营销官网风格
3. 普通用户区更偏功能可用，还没有做强运营化包装
4. 多平台内容源还在继续扩展中

这几个边界可以直说，不需要回避。
